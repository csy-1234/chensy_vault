
Stability First
	任何操作都不能影响线上服务稳定性。
		不在高峰期做变更；有计划、审批的变更；操作必须可回滚；不能在生产环境直接“试试看

Minimal Change
	变化要小，减少对系统的影响
		只改必须改的；一次变更不要覆盖太多内容；测试环境验证后再到生产环境执行
Rollback
	上线或变更前明确好如何撤销、如何回退
		代码版本控制；备份配置；数据快照；
Automation
	重复性、易出错的流程必须自动化
		CI/CD（自动化部署）；自动化巡检；自动化报警；自动化备份
监控
	CPU / 内存 / 磁盘；服务状态、接口耗时；错误日志；业务指标（订单数、流量、转化率）；告警准确性；
Security
	密码加密；密钥保密；least privilege；防火墙、WAF、访问控制；补丁、升级安全版本；
灾备
	数据库；文件；配置；异地灾备；定期演练；
文档化
	规范文档
		变更记录
		sop（操作手册）standdard operating procedure
		架构图、网络图
		系统说明
权限与分权
	root生产环境禁止登陆
		禁止root ssh登录；使用sudo；
	DBA、开发、运维权限分开；
	Audit，审计与记录（日志、堡垒机），建立追责机制
	每个人有自己的账号




运维故障与功能故障：

	运维故障即 运维操作、部署、配置、资源、网络等引发的生产系统异常，代码本身没有问题。这些都是运维行为或环境引起的故障，核心业务可能受阻，甚至无法访问，需要快速响应，需要分钟级或小时级快速响应
	功能故障：代码逻辑、设计、功能不完善，功能故障不影响核心业务，响应时间是天级的；
	
    功能缺陷可以先在测试环境发现；
    运维故障只在生产环境出现。

 


响应时间：
	问题产生产生到人员开始处理的时间。

运维故障响应、处理时间限制：

| 优先级       | 描述        | 典型示例            | 响应时间       | 处理时间          |
| --------- | --------- | --------------- | ---------- | ------------- |
| P1 / Sev1 | 核心业务完全不可用 | 核心服务宕机、支付接口不可用  | ≤ 5 分钟     | ≤ 30 分钟至 1 小时 |
| P2 / Sev2 | 关键业务部分受影响 | 部分接口异常、部分用户无法访问 | ≤ 15–30 分钟 | ≤ 2–4 小时      |
| P3 / Sev3 | 次要功能异常    | 辅助模块异常、非核心服务    | ≤ 1 小时     | ≤ 1 工作日       |
| P4 / Sev4 | 轻微异常或优化   | 监控告警、日志异常       | 当班响应       | 按排期解决         |
> **响应时间通常在分钟级，处理时间通常在小时级，必须立即介入处理，保证业务连续性。**

功能故障响应、处理时间限制：

|优先级|描述|响应时间（确认问题并安排处理）|解决/修复时间（上线）|
|---|---|---|---|
|高（Critical / P1）|核心功能受影响，业务可能受阻|1 个工作日内|1–3 个工作日内修复或上线临时方案|
|中（Major / P2）|关键功能异常，但不阻塞核心业务|2 个工作日内|1–2 周内修复或纳入下个版本|
|低（Minor / P3）|功能可用，但存在体验或非关键问题|3 个工作日内|下个版本迭代修复|
|建议 / Enhancement / P4|优化、改进或可选功能|可记录后评估|按产品计划安排|
功能故障不如运维故障紧急，但仍需规范化响应：**确认→分配→排期→修复→验证→通知**，响应时间通常是 **工作日级**，解决时间按版本或迭代安排。
