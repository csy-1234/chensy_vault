一.
- **`docker exec` 不会读取 Entrypoint 和 CMD**
    
    - 它只是在 **已有容器** 内启动一个新进程
        
    - 容器已经在运行，Entrypoint 和 CMD 在容器启动时已经执行过了
        
- **必须指定命令**
    
    - 因为 `docker exec` 不会默认启动任何东西，它需要你告诉它“启动什么进程”
        
    - 典型用法：
        
        `docker exec -it <container> /bin/bash docker exec -it <container> sh docker exec -it <container> ps aux`
        
    - 如果不指定命令，Docker 会报错：
        
        `Error: no command specified`

二、

docker bridge 网络
	`bridge` 网络在宿主机上是一个虚拟的局域网
	宿主机是这个虚拟网的网关（外网流量走宿主机）
	IP 与容器一一对应
	容器之间可以互相访问
	端口映射，部访问容器必须有端口映射
docker容器访问的三种方式：
	容器之间可以互相访问：ip或容器名：端口；
	宿主机本身可以直接访问容器的 bridge 内网 IP
	宿主机外的设备访问容器，必须通过端口映射（bridge模式）；（host模式不用端口映射，容器间通信必须通过宿主机 IP + 端口）

### git

git push推送
```
push指定本地分支到远程分支：
把你本地 dev4 的新提交（新commit）推到远程，本地提交树追加到远程的提交树中
把远程指针指向你的最新版本
总结：更新远程仓库，让远程分支指向你的最新提交

格式：
git push <远程仓库名> <本地分支>:<远程分支>

情况 0：首次推送并建立追踪关系
git push -u origin <本地分支>:<远程分支>（同名写一个）
情况 1：你当前就在 test 分支
git switch test
git push
结果：
👉 会推到远程的 origin/test
（前提：本地 test 与 origin/test 已建立 tracking 关系）
情况 2：你明确 push test 分支
git push origin test
结果：
👉 永远推到远程的 origin/test
如果远程没有这个分支，Git 会自动创建

失败判断：
提交树本地大于远程实际push追加新的提交，如果远程多了本地没有的commit，则失败

远程协作同分支gitpush失败，解决：
git fetch +git merge o/main=pull
或rebase


建立追踪关系：
git push -u origin dev4
或
git branch --set-upstream-to=origin/dev4 dev4
或


没有追踪关系时
你必须显式写：
git push origin dev4
git pull origin dev4

git clone：建立远程分支引用origin/main

git branch -vv

查看远程仓库别名及 URL：
```
